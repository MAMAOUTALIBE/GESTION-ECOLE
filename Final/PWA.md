# GESTION-EE PWA (Module 16)

This document describes how the Progressive Web App layer of the GESTION-EE
front-end is wired, how to test it locally and what its limits are. The PWA is
purely **additive** ‚Äî it does not change the Spruko design.

## What is shipped

- **`@angular/service-worker`** Angular SW (`ngsw-worker.js`), generated by
  the CLI from `ngsw-config.json`.
- **`public/manifest.webmanifest`** with `name`, `theme_color`
  (`#1d6b3a`, the GESTION-EE green), `start_url`, `display: standalone`
  and three placeholder PNG icons under `public/assets/pwa/`.
- **Cache strategies** (see `ngsw-config.json`):
  - `app` (assetGroup, `prefetch`) ‚Äî `index.html`, `*.css`, `*.js`,
    `manifest.webmanifest`, `favicon.ico`. Installed on first visit.
  - `assets` (assetGroup, `lazy` / `updateMode: prefetch`) ‚Äî everything
    under `/assets/**` plus all common image / font extensions.
  - `api-cartography` (dataGroup, `freshness`, 10s timeout, 1 day TTL,
    100 entries) ‚Äî `/api/cartography/**`.
  - `api-dashboard` (dataGroup, `freshness`, 5s timeout, 1 hour TTL,
    50 entries) ‚Äî `/api/census/dashboard` and `/api/analytics/**`.
  - `api-opendata` (dataGroup, `performance`, 5s timeout, 1 day TTL,
    30 entries) ‚Äî `/api/opendata/**`.
- **`NetworkService`** (`src/app/shared/network.service.ts`) ‚Äî exposes
  `online$: Observable<boolean>` and `isOnline(): boolean`. Listens to
  `window.online`/`offline` events.
- **`OfflineBannerComponent`** (standalone) ‚Äî fixed red bar at the top of
  the viewport, only rendered when offline. Wired in `app.html` so it covers
  the auth layout as well as the content layout.
- **`OfflineQueueService`** ‚Äî IndexedDB-backed scaffold for queueing
  write requests while offline. **Not auto-flushed.** Module 17 (mobile sync)
  is expected to call `flush()` once auth state is ready.

## Why `@angular/service-worker` rather than a custom Workbox build

- Already supported by `@angular/build` ‚Äî declarative `ngsw-config.json`
  beats hand-rolled Workbox boilerplate.
- Built-in update lifecycle via `SwUpdate` (versionned cache, hash-based
  invalidation, `versionUpdates$`).
- Zero impact on the dev workflow ‚Äî registration is gated on
  `!isDevMode()` in `app.config.ts`, so `npm start` keeps hot-reload.

## How to test locally

### 1. Build a production bundle

```bash
cd Final
npm run build:node20
```

The output goes to `Final/preview/` and must now contain:

- `ngsw.json` ‚Äî the runtime manifest the SW reads at boot
- `ngsw-worker.js` ‚Äî the service worker script itself
- `manifest.webmanifest` ‚Äî the web app manifest
- `assets/pwa/icon-192.png`, `icon-512.png`, `icon-maskable.png`

### 2. Serve over HTTP (NOT `file://`)

Service workers only register over HTTPS or `http://localhost`. The
simplest local server:

```bash
cd Final/preview
npx http-server -p 8080 -c-1
```

then open <http://localhost:8080/>.

> Note: `baseHref` defaults to `/angular/nowa/preview/` in the production
> configuration. If you serve `preview/` at the root, you may want to build
> with `--base-href /` for local testing only.

### 3. Verify in DevTools

In Chrome / Edge DevTools:

- **Application > Service Workers** ‚Äî `ngsw-worker.js` should be `activated and running`.
- **Application > Manifest** ‚Äî should list "GESTION-EE ‚Äî Pilotage scolaire national".
- **Application > Cache Storage** ‚Äî at least `ngsw:...:app:cache` and `ngsw:...:assets:cache`.
- **Network** tab ‚Äî flip "Offline" and reload: the shell still renders,
  the red banner appears, cached APIs return cached responses.

### 4. Trigger an update

Service workers are versionned by the hash of `ngsw.json`. Whenever you
ship a new build, the SW downloads the new version in the background and
activates it on the next navigation. To force the update during testing:

- DevTools > Application > Service Workers > **Update on reload**, then F5.
- Or call `SwUpdate.checkForUpdate()` from a UI affordance (not wired in
  the MVP ‚Äî see backlog 16.1).

## Limits / known caveats

- **Icons are placeholders.** `icon-192/512/maskable.png` are solid-green
  squares generated by a tiny PNG encoder in the build pipeline. Replace
  them with branded artwork before going to production.
- **Offline queue is a scaffold.** `OfflineQueueService` persists request
  descriptors but never auto-flushes. There is no UI to inspect or replay
  pending requests yet. Module 17 (mobile) will own this.
- **No background sync.** We rely on Module 17 to call `flush()` explicitly.
- **Auth tokens are NOT stored** in the offline queue. If a request is
  queued offline, the eventual replay needs to re-attach the current auth
  header via an interceptor.
- **First load still requires network.** All `assetGroups` are `prefetch`
  or `lazy` ‚Äî the very first visit must hit the network to seed the cache.
- **Cross-origin `fetch`** (e.g. Google Maps, Google Fonts) is not handled
  by the SW; those resources will fail offline. This is by design.
